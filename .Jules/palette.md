## 2024-05-24 - Accessibility handling for inline form validation

**Learning:** When using vanilla JavaScript for inline form validation (e.g., dynamically displaying an error `<small>` element on input change), using standard visual cues like `display: block` or styling in red is insufficient for screen readers. Explicitly linking the input to the error message container using `aria-describedby` and dynamically toggling `aria-invalid="true"`/removing `aria-invalid` provides immediate context. Furthermore, making the error container `aria-live="polite"` ensures that screen readers dynamically announce the error message when it appears without interrupting the user unnecessarily. Finally, when querying DOM elements for these updates in a vanilla environment, relying on implicit global variables created by HTML element IDs (e.g., `erro.style.display` where `id="erro"`) can silently break or cause `ReferenceError`s; explicit `document.getElementById` must be used.

**Action:** Whenever implementing custom form validation logic, always establish an `aria-describedby` relationship between the input and its error message container. Use an `aria-live` region for the error container and ensure the script dynamically toggles the `aria-invalid` attribute on the input field alongside visual state changes. Always explicitly select DOM elements in JavaScript.
